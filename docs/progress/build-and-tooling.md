# Build system, CI, and tooling

Covers the `Makefile`, `.github/workflows/ci.yml`, and repo-level recovery -
not any one `frelsim/` subdirectory, but the scaffolding everything else
depends on.

## Repo recovery (project start)

**Starting state.** The local git repository had ~59 corrupted (zero-byte)
loose objects, including the object the local `main` HEAD pointed to - `git
status`/`log`/`fsck` all failed outright. The working tree itself was a
third, distinct state: a mix of uncommitted local edits sitting on top of
the last valid local commit, while `origin/main` had several commits
further ahead that were never pulled down (a more advanced `EventEngine`, an
abstracted `Model` base class). None of the three states (corrupted local
HEAD, uncommitted local edits, origin/main) matched each other, and none of
them compiled.

**Recovery.** Confirmed via `git fetch` (after clearing the corrupted loose
objects so fetch could complete) that `origin/main` was a strict, unbroken
descendant of the last valid local commit - nothing was actually lost. The
uncommitted local edits were committed as a snapshot and preserved on a
`local-wip-before-reconcile` branch (untouched, available if anything in it
turns out to be worth salvaging), and `main` was fast-forwarded to
`origin/main` as the base to build forward from.

## Makefile

**Getting to a compiling build.** `Model` had already been correctly made
abstract in an earlier commit (subclasses must implement `update`/
`derivative`/`getOutputs`/`setInputs`), but `ModelFactory` and `ModelAdapter`
had not been updated to match - `ModelFactory` had the wrong signature and an
empty body, and `ModelAdapter` tried to instantiate `Model` directly (illegal,
it's abstract) and reached into members that had been left `private` with no
way for a subclass to reach them. Fixed the whole chain: `ModelFactory`
dispatches on `component_name`, `ModelAdapter` routes through it, and `Model`'s
state vectors (`continuousStates_`, `discreteStates_`, `inputs_`, `outputs_`,
`parameters_`) were moved from `private` to `protected` so subclasses can
actually read/write them.

**Toolchain.** Found and fixed a build-system bug where gRPC service stubs
were being generated by `protoc` but never actually linked into
`libfrelsim.a` - a `$(wildcard ...)` used to discover the generated
`.grpc.pb.cc` files was evaluated before codegen ran (so it always saw an
empty list), and separately the Makefile's pattern rules expected
`_grpc.pb.cc` filenames while `protoc` actually emits `.grpc.pb.cc`. Also
discovered two separate, ABI-incompatible protobuf/gRPC installs on the build
machine (a system apt package, and a separately-built newer one on `PATH`
ahead of it) - resolved by installing the apt `protobuf-compiler-grpc` package
(version-matched to the already-installed system `libgrpc++-dev`) and pointing
the Makefile at `/usr/bin/grpc_cpp_plugin` explicitly rather than trusting
`PATH` order. Added `-MMD -MP` header-dependency tracking, since without it
editing a `.hpp` did not invalidate the `.o` files that transitively include
it.

**The `--whole-archive` self-registration gotcha.** `Model`/`Solver`
subclasses self-register via a static initializer in their own `.cpp`
(`FRELSIM_REGISTER_MODEL`/`FRELSIM_REGISTER_SOLVER` - see
[`models.md`](models.md) and [`solvers.md`](solvers.md)), specifically so
`ModelFactory`/`SolverFactory` never have to `#include` every concrete
subclass. Because `libfrelsim` is a static archive (`.a`), `ld` only pulls
an archive member in to resolve an outstanding undefined symbol - since
nothing else references a model's/solver's `.o` once the factory doesn't,
a plain `ar`-built archive link would silently drop it and its registration
would just never run (`createModel("SomeModel")` returns `nullptr` at
runtime, no error, no warning). Every executable that needs the registries
populated (`frelsim_tests`, `frelsim_sim`, any future one) must link
`libfrelsim.a` via `-Wl,--whole-archive ... -Wl,--no-whole-archive` - the
Makefile's `LINK_LIBFRELSIM` variable does this; use it (or match it
exactly) rather than linking `$(TARGET)` directly.

**`sim` target** (added alongside `sim/main.cpp` - see
[`sim-executable.md`](sim-executable.md)): `SIM_DIR`/`SIM_SRC`/`SIM_BIN`
mirror the existing `test` target's pattern, linked through the same
`LINK_LIBFRELSIM` for the same self-registration reason.

## Testing infrastructure

Previously: an ad-hoc, uncommitted driver program in the scratch directory,
used to manually confirm `BouncingBall`'s behavior. Not checked in, not
repeatable via any single command.

Now: a real `test/` package using GoogleTest, wired into the Makefile as
`make test`, mirroring `frelsim/`'s directory layout file-for-file. Every
component's specific test additions (and the regression tests among them)
are documented alongside that component in its own file under
`docs/progress/`, not repeated here.

**GMock**: `libgtest-dev` was already installed system-wide; the matching
`libgmock-dev` is not yet installed and requires `sudo`, which isn't
available in this dev environment. The Makefile detects it via
`pkg-config` and gracefully skips any test file with `Mock` in its name
until it's present, so `make test` works either way. No tests currently
need it - everything tested so far is concrete and state-based (no
interface needs mocking). The natural place GMock would earn its keep is
testing `Simulator` orchestration against a mocked `SimAdapter`/`Model`.

## CI

`.github/workflows/ci.yml` runs on push to `main` and on every PR:
installs `protobuf-compiler`, `libprotobuf-dev`, `libgrpc++-dev`,
`protobuf-compiler-grpc`, `libeigen3-dev`, `libgtest-dev`, `libgmock-dev` via
apt, then `make` and `make test`. Once `sim/main.cpp` existed, also added
`make sim` plus a run of `./build/frelsim_sim` against
`examples/pid_mass_spring_damper.json`, so the executable runner path stays
covered on every push/PR too.

## Environment quirks (this dev machine specifically)

- Multiple gRPC/protobuf installs exist (system apt packages vs. a
  separately-built newer one). The Makefile pins `grpc_cpp_plugin` to
  `/usr/bin/grpc_cpp_plugin` explicitly and derives `pkg-config` search
  paths from wherever the plugin actually resolves - if gRPC-related build
  errors mention symbol/ABI mismatches, check `GRPC_CPP_PLUGIN`/
  `GRPC_PREFIX` in the Makefile before assuming the code is wrong.
- `libgmock-dev` is not installed, and there's no `sudo` available - if
  GMock-based tests are ever needed, the user has to install it themselves.
- `gh` CLI is not installed - PRs are opened by pushing the branch and
  handing the user the `.../pull/new/<branch>` URL git prints on push.
- A clean build has occasionally hit a transient GCC internal compiler
  error (ICE) that didn't reproduce on retry - if a build fails with an ICE
  ("Please submit a full bug report"), retry once before assuming the code
  is at fault.
- `make`/`make test` are slow (~1-3 min, mostly linking the
  gRPC/abseil-heavy test binary) - run them with a generous timeout rather
  than assuming a short one is enough.
